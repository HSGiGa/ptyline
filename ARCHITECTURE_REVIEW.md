# Architecture Review — ptyline

Дата: 2026-07-02. Дополнение к [CODE_REVIEW.md](CODE_REVIEW.md): там —
конкретные дефекты, здесь — структурные свойства, которые эти дефекты
порождают. main @ 06d7026, ~15K строк Go, 25 внутренних пакетов.

## Резюме

Архитектура в целом зрелая и соответствует своему описанию в
ARCHITECTURE.md: однонаправленный поток данных, единственный мутатор
состояния (event loop), один сериализованный writer, типизированный sealed
event set, вынесенная в `reserved` математика «rows − reserved»,
capability-based платформенная модель. Граф зависимостей пакетов ациклен и
корректно слоится (утилиты → домен → wiring); `go list` не показывает ни
одного цикла или «протечки» вверх.

Главные структурные риски — не в границах пакетов, а внутри `internal/app`:
композиционный корень разросся в 700-строчную функцию с тремя разными
механизмами жизненного цикла модулей, конфиг «запекается» в конструкторы
(корневая причина reload-багов), а слой наблюдаемости существует только как
каркас — ошибки глотаются молча.

---

## A1. `app.run()` — 697-строчный композиционный корень с логикой в замыканиях

`internal/app/app.go:81-777`

`run()` совмещает три роли: парсинг/детект окружения, сборку пайплайна и
**управление жизненным циклом модулей** (`initModules`, `updateModules`,
`startUserMod`, `reloadConfig`, `restartTicker`, `refreshGit*` — всё это
замыкания, разделяющие ~20 мутабельных захваченных переменных:
`resolvedCfg`, `area`, `barRows`, `render`, `visuals`, `gitMod`,
`timeCancel`/`dateCancel`/`gitCancel`/`tickerCancel`, `userMods`,
`execModules`, `shellStyles`, `resizePending`…).

Последствия:

- **Нетестируемость**: `app_test.go` покрывает только хелперы и
  `execModuleRuntime`; сам reload/resize/alt-screen wiring — самая
  насыщенная логикой часть приложения — не покрыт ни одним тестом, потому
  что его нельзя сконструировать вне `run()`.
- **Хрупкость reload**: инварианты вида «`refreshGit` можно звать только из
  loop-горутины, а тикер должен захватывать `gm` по значению» существуют
  как комментарии, а не как типы/структура.

Рекомендация: выделить `type appState struct` (или несколько: `barState`,
`moduleSet`) с методами, конструируемый в `run()` и тестируемый отдельно.
Это механическая, поэтапная работа — замыкания уже группируются в очевидные
кластеры.

## A2. Конфиг модулей запекается в конструкторы → reload как «diff-and-restart» в четырёх экземплярах

Каждый тип модуля решает задачу «применить новый конфиг» по-своему:

| Механизм | Где | Что сравнивает |
|---|---|---|
| built-in time/date | app.go:421-446 | format + interval, вручную ×2 |
| git | app.go:449-462 + `Git.SameConfig` | interval/format/separator/maxWidth |
| user (exec/time) | app.go:469-502 | **только** command + interval |
| probe-модули | probemodules.go:98-114 | interval + format |

Четыре несовпадающих подмножества полей — и это уже дало баг (CODE_REVIEW
H3: exec-модули игнорируют изменения format/env/timeout при reload), а
`cmdTracker` вообще не участвует в reload. Каждый новый параметр модуля
требует не забыть дополнить нужный из четырёх diff'ов — архитектура
гарантирует, что когда-нибудь забудут снова.

Два системных выхода:

1. **Обобщить паттерн `probeModManager`** (он самый полный: probe +
   schedule + reconcile) на все типы модулей — один reconciler, один
   контракт «модуль перезапускается, если изменился его `ConfigKey()`»,
   где ключ — сравнение всего релевантного среза конфига (например,
   сериализованный `ModuleConfig`), а не выбранных вручную полей.
2. Либо инвертировать: модуль читает конфиг атомарно на каждом refresh
   (`atomic.Pointer[ModuleConfig]`), и рестарт нужен только при смене
   interval.

## A3. Alt-screen состояние хранится в трёх местах

`AnsiFilter.alt.Active` (ansi_filter.go:50), `TerminalWriter.altActive`
(writer.go:30), `StatusState.Terminal.AlternateScreen` (state.go:83) —
синхронизируются вручную через `altScreenCoordinator.Apply` и через
`filter.AltActive()` в `ResizeCommit`. Сейчас все переходы идут через
координатор и loop-горутину, так что рассинхрон маловероятен, но инвариант
«три копии всегда равны» нигде не выражен и не проверяется. Достаточно
одного владельца (фильтр — он единственный, кто детектит переход) и
читателей через него; writer'у флаг можно передавать параметром кадра.

Смежное: `cwdHolder` (atomic.Value) дублирует `state.Shell.CWD` ради
кросс-горутинного чтения. Паттерн правильный, но неявный; стоит оформить
его как явный тип (`type SharedCWD struct{ atomic.Value }`), чтобы правило
«StatusState — только для loop; для горутин — атомики» читалось из кода.

## A4. Наблюдаемость: каркас есть, подключения нет — ошибки исчезают молча

- `AnsiFilter.SetDiagHandler` **никогда не вызывается** в продакшн-коде —
  диагностика об oversized/malformed последовательностях и отброшенных
  OSC 777 теряется (grep по репо: только определение и тесты).
- `internal/diagnostics.State` не получает ни одной записи;
  `StatusState.Diagnostics` — мёртвое поле.
- Ошибки reload проглатываются без следа (app.go:528,740; CODE_REVIEW M3);
  `VisualsFromConfig` при reload тоже (app.go:546 — `if err == nil`).

Для TUI-обёртки stderr во время работы недоступен — значит, архитектурно
нужен *какой-то* канал: env-gated debug-лог в файл
(`PTYLINE_DEBUG=path`), заполнение `diagnostics.State` + индикатор в баре,
и `doctor`/`debug-state` CLI (уже заявлен как deferred). Пока канала нет,
каждая «мягкая» деградация (keep running on bad config) — это молчаливое
враньё пользователю. Это дешевле сделать до того, как появятся баг-репорты
вида «бар просто не обновился».

## A5. Пирамида тестов без вершины: инварианты терминала не проверяются end-to-end

Юнит-покрытие хорошее (фильтр, writer, layout, renderer, config), но:

- **Нет PTY-харнесса**: ни один тест не запускает настоящий child в PTY и
  не проверяет главные инварианты (§8 spec: бар никогда не скроллится,
  scroll region восстановлен, exit code проброшен) на реальном байтовом
  потоке. Именно на этом уровне живут пропущенные дефекты (RIS из H4 —
  сценарий `reset`).
- **Шаблоны shell-интеграции тестируются grep'ом строк**
  (registry_test.go: «скрипт содержит `exit_code`»), а не исполнением.
  Поэтому `date +%s%3N` (H1) — синтаксически валидный, семантически
  сломанный на macOS — прошёл все тесты. Нужен таблично-управляемый тест,
  который запускает каждый доступный шелл (`bash -c`, `zsh -c`,
  `fish -c`) с загруженным скриптом и парсит эмитированные OSC-фреймы;
  скипается, если шелла нет в PATH.
- Рекомендуемая вершина: golden-transcript тест — скармливать записанные
  байтовые потоки (vim, top, fish multiline, `reset`, resize) через
  filter+writer в мини-эмулятор (например `vt10x`) и ассертить содержимое
  зарезервированных строк.

## A6. Три shell-шаблона дублируют одну логику и уже разъехались

`templates/{bash,zsh,fish}.sh` независимо реализуют: валидацию имён
переменных, emit_env (с одинаковой странной веткой `count == 1`),
exec_env-протокол (base64 + nonce), ssh-обёртку, тайминг. Дрейф уже
случился: fish использует `$EPOCHREALTIME`, bash/zsh — сломанный на macOS
`date +%s%3N` (H1); fish эмитит `colors` и cwd-хук на `--on-variable PWD`,
остальные — нет.

«Adding a shell is a template file, never Go logic» — правильный принцип,
но он не требует, чтобы шаблоны были рукописными монолитами. Варианты по
возрастанию усилий: (а) общий контрактный тест из A5, фиксирующий
одинаковую семантику фреймов; (б) генерация шаблонов из общего описания
протокола + per-shell идиомы. Минимум — (а).

## A7. `event.ModuleUpdated.Snapshot any` — обход цикла импортов вместо его устранения

`internal/event/event.go:44-47`: payload типизирован как `any`, потому что
`event` не может импортировать `status`. Loop делает type assertion
(app.go:721-724), молча игнорируя неудачу. Цикл возникает из-за того, что
`status` — одновременно и модель данных, и фреймворк модулей. Если вынести
`ModuleSnapshot`/`ModuleID` в листовой пакет (или в сам `event`),
assertion исчезает и sealed-set становится по-настоящему типизированным.
Не срочно, но каждый новый тип события с payload будет воспроизводить
этот компромисс.

## A8. `config` зависит от `status/layout`

`internal/config/loader.go:11` — валидация шаблонных модулей вызывает
`layout.ParseFormat`. Доменный слой конфигурации зависит от UI-пакета
раскладки; кроме того, «как парсится format-строка» теперь неявно является
частью контракта конфига. Парсер format-строк — самостоятельная вещь
(лексер плейсхолдеров) и естественно живёт в отдельном листовом пакете
(`internal/format`), от которого зависят и `layout`, и `config`.

## A9. Renderer пересоздаётся на каждый resize/reload и теряет своё состояние

app.go:549,613: `render = renderer.New(...)` при каждом `ResizeCommit` и
reload. Renderer несёт change-detection состояние анимаций (`lastValues`,
`changedAt`) — оно обнуляется, т.е. resize посреди «glint»-анимации её
обрывает/перезапускает. Косметика, но симптом смешения: у renderer'а два
вида состояния — конфигурационное (theme/styles/templates, пересоздавать
правильно) и рантаймовое (фазы анимаций, должно переживать resize). Их
стоит разделить (engine пересоздаётся, animation-tracker живёт отдельно).

## A10. Единая очередь для PTY-байтов и контрольных событий

Все источники — включая высокочастотный `PtyOutput` (32 КБ чанки) —
мультиплексируются в один канал (буфер 256) с блокирующим backpressure.
Это осознанный и задокументированный выбор (упорядоченность важнее
латентности), но у него есть следствие, которое стоит держать в голове при
добавлении Agents/mouse: при заливке вывода (например, `cat` большого
файла) `Tick`/`ModuleUpdated`/`ConfigReloadRequested` ждут в общей очереди,
а producer'ы модулей блокируются на `SendCtx`. Пока событий мало — норм;
приоритетная очередь или отдельный канал для контрольных событий — это
изменение, которое лучше заложить до Agents-инжеста, а не после.

## A11. Миграция конфига — regex по тексту

`internal/config/migrate.go:32`: `config_version = 0 → 1` через
`regexp.ReplaceAll` по сырому TOML. Для одного шага работает, но модель «N
regex-переписываний сырого текста» не масштабируется (комментарии, инлайн-
таблицы, нестандартное форматирование). Правильная форма при появлении
версии 2: decode в per-version структуру → программная трансформация →
merge, без текстовых подстановок.

## Мелкое

- `internal/command` — название пакета конфликтует по смыслу с
  `module.command` и `exec`-провайдером «command»; сам пакет — app-уровневая
  политика отображения. Точнее было бы `internal/cmdtracker` или подпакет app.
- `Makefile` переопределяет `GOCACHE` в каталог репозитория — локальные
  сборки не разделяют кэш с остальной системой (медленнее первый билд;
  вероятно, сознательно для CI, но задевает и разработчика).
- `status.GitState` (state.go:114-118) помечен как «reserved», но git
  провайдер уже реализован через модульные снапшоты — поле, судя по всему,
  уже никогда не будет использовано в заявленном виде; лучше удалить, чем
  резервировать.
- `AnimationPhase int` инкрементируется без сброса (app.go:727) —
  практически не переполнится, но `%`-использование в renderer'е сделало бы
  это очевидным.

## Что уже хорошо (и стоит сохранять)

- Ациклический, слоистый граф пакетов; `app` — единственное место сборки.
- `reserved.Area` как единственный источник правды для геометрии.
- Sealed event set + единственный мутатор состояния; правило «renderer
  никогда не опрашивает провайдеров» выдержано во всём коде.
- `probeModManager` — образцовый lifecycle-паттерн (probe → schedule →
  reconcile, initial refresh off-loop); его и стоит распространить (A2).
- Shell-агностичный Go + data-driven реестр шаблонов.
- Дисциплина таймаутов: ни одного внешнего вызова без deadline;
  `WaitDelay`/process-group у exec — редкая аккуратность.
- Оверлеи с запретом command-полей — правильная граница доверия.

## Приоритеты

1. **A4** — подключить диагностику (дёшево, разблокирует отладку всего
   остального).
2. **A5** — исполняемые тесты шаблонов + golden-transcript харнесс (ловит
   класс ошибок H1/H4 до релиза).
3. **A2** — единый reconciler конфигурации модулей (закрывает H3 как класс).
4. **A1** — декомпозиция `run()` (делает 1–3 проще; можно вести поэтапно).
5. A3/A7/A8/A9 — по мере касания соответствующего кода.
