# Code Review — ptyline

Дата: 2026-07-02. Ревью всей кодовой базы (main @ 06d7026, ~15K строк Go).
`go vet ./...` — чисто; `go test ./...` — один падающий тест (см. H5).

Общее впечатление: архитектура продуманная и дисциплинированная — один
сериализованный writer, единственный мутатор состояния (event loop),
таймауты на всех внешних вызовах, nonce для exec_env, запрет команд в
project-оверлеях. Найденные слабые места ниже, по убыванию серьёзности.

## Высокая серьёзность

### H1. `date +%s%3N` ломает zsh/bash-интеграцию на macOS
`internal/shellintegration/templates/zsh.sh:15`, `templates/bash.sh:15`

`__ptyline_now_ms() { date +%s%3N; }` — `%3N` это GNU-расширение; BSD `date`
на macOS выводит литеральные символы: `17829644253N`. Дальше арифметика
`$(( now - start ))` падает:

```
$ zsh -c 'a=$(date +%s%3N); b=$(date +%s%3N); echo $((b - a))'
zsh:1: bad math expression: operator expected at `N'
```

Итог: на основной целевой платформе (macOS, где zsh — шелл по умолчанию)
после каждой команды в терминал печатается ошибка, а `duration_ms` не
эмитится вовсе. fish-шаблон уже решает это через `$EPOCHREALTIME` — zsh
может использовать тот же `$EPOCHREALTIME` (zmodload zsh/datetime), bash ≥5
имеет `$EPOCHREALTIME` тоже.

### H2. bash: DEBUG-trap ловит чужие записи PROMPT_COMMAND → фантомная «активная команда»
`internal/shellintegration/templates/bash.sh:76-108`

DEBUG-trap срабатывает и для команд внутри `PROMPT_COMMAND`. Гард
пропускает только `__ptyline_precmd*`. Если у пользователя
`PROMPT_COMMAND="…;history -a"` (типично: bash-preexec, iTerm2, starship):

1. после каждого промпта `history -a` эмитится как активная команда —
   в баре висит фантом;
2. `__ptyline_running=1` остаётся выставленным, поэтому следующая
   настоящая команда пользователя **не** репортится (ранний return);
3. duration считается от старта фантома и включает время простоя у промпта.

Дополнительно `trap '__ptyline_preexec' DEBUG` затирает существующий
DEBUG-trap пользователя (ломает bash-preexec и всё, что на нём построено).
Нужен гард «мы внутри PROMPT_COMMAND» (как в bash-preexec) и дописывание к
существующему trap, а не замена.

### H3. Reload не применяет большинство изменений exec-модулей и модуля command
`internal/app/app.go:486-502`, `internal/app/app.go:161`

`updateModules` перезапускает exec-модуль только при смене `command` или
`interval`. Но `format`, `separator`, `max_width`, `timeout_ms`, `env`,
`refresh_on_command`, `refresh_on_cwd` зашиты внутрь `modules.Exec` /
`execModuleRuntime` при создании. Комментарий «format changes are picked up
via configureRenderer (templates)» неверен для exec: template-спеки
покрывают только `source = "template"`, а формат exec-модуля подставляется
в `Exec.RefreshWithEnv` (`internal/modules/exec.go:137-141`). Изменение
этих полей в конфиге + `ptyline --reload` молча не действует.

Аналогично `cmdTracker` создаётся один раз из стартового конфига и не имеет
сеттера: изменения `module.command.*` (TTL, format, max_width,
active_min_duration_ms) при reload игнорируются до перезапуска ptyline.

### H4. Фильтр не перехватывает RIS (`ESC c`) и DECSTR (`CSI ! p`) — защита бара теряется после `reset`
`internal/proxy/ansi_filter.go:262-291`

Оба сброса терминала обнуляют scroll region (а RIS ещё и чистит весь
экран, включая зарезервированные строки). Фильтр их не обрабатывает,
и никто не переустанавливает scroll region после них — до следующего
resize или alt-screen перехода дочерний вывод свободно скроллит по строкам
бара, т.е. нарушается инвариант №2 из ARCHITECTURE.md. Ребёнок легко
эмитит RIS реальной командой `reset` / `tput reset`.

### H5. Падающий тест: `TestExecRefreshWithDir`
`internal/modules/exec_test.go:141-160`

`NewExec(…, maxWidth=0)` включает дефолтный кап 60 символов
(`defaultExecMaxWidth`), а путь `t.TempDir()` на macOS
(`/private/var/folders/…/TestExecRefreshWithDir…/001`) длиннее 60 —
вывод `pwd` обрезается с многоточием и сравнение падает. Тест зависит от
окружения (длины TMPDIR и имени теста). Нужно передавать явный большой
`maxWidth` в тесте (или сверять с усечённым ожиданием).

## Средняя серьёзность

### M1. Reload блокирует event loop синхронными запусками команд
`internal/app/app.go:357` (`startUserMod` → `em.refreshSnapshot(context.Background())`),
`internal/app/app.go:459` (`gitMod.RefreshAll(context.Background())` в `updateModules`)

Оба вызова выполняются в горутине event loop при ConfigReload/смене
project-оверлея и синхронно ждут внешнюю команду (до её таймаута, по
умолчанию 1 с; git — 1 с). На это время замирает весь ввод/вывод терминала;
с несколькими новыми exec-модулями — кратно. Первичный снапшот стоит
получать off-loop, как это уже делает `probeModManager.start` через
`scheduler.RefreshOnce` (internal/app/probemodules.go:132-133).

### M2. Результат наложения оверлеев никогда не валидируется
`internal/config/loader.go:552-585`

`Load` валидирует базовый конфиг, но `ApplyOverlays` (и стартовый путь в
`run()`) не вызывает `Validate` на смёрженном результате. Оверлей может
протащить невалидные enum-значения (`bar.justify`, `icons.preset`,
`animation`) и, главное, экстремальный `interval_ms` (например 1 мс —
запуск пользовательской команды 1000 раз/с). Нижней границы интервала нет
нигде (`moduleInterval` отбрасывает только `<= 0`). Стоит валидировать
merged-конфиг и ввести минимальный интервал для exec/git.

### M3. Ошибки reload молчаливые, а сломанный `.ptyline` не перечитывается после исправления
`internal/app/app.go:521-531`

`reloadConfig` сохраняет `projectOverlayPath` **до** парсинга; при ошибке
парсинга возвращает `false` без какой-либо диагностики. Последствия:
пользователь не видит причину (нет ни записи в diagnostics, ни stderr), а
из-за уже сохранённого пути guard `old == newProjectPath` не даст повторить
попытку для того же файла — исправление `.ptyline` не подхватится при cd
внутри проекта, только явный `ptyline --reload`.

### M4. Аутентифицирован только `exec_env`; `cwd`/`env`/`command`/`exit_code` подделываемы
`internal/proxy/osc.go`, `internal/app/app.go:642-703`

Nonce защищает лишь exec_env-фреймы. Любая программа, способная вывести
байты в терминал (в т.ч. `cat` вредоносного файла), может эмитить
`OSC 777;cwd=/attacker/path`: это (1) меняет `cmd.Dir` для всех
exec-модулей — пользовательские команды выполняются в каталоге атакующего;
(2) запускает поиск и применение project `.ptyline` по указанному пути;
(3) `exit_code=0` триггерит `refresh_on_command`-обновления. Модель угроз,
из-за которой добавлен nonce (инъекция байтов в поток), ровно так же
применима к `cwd`. Минимум — распространить nonce на все ключи, влияющие
на исполнение (cwd), а не только на env.

### M5. `--reload` шлёт SIGUSR1 непроверенному PID из окружения
`internal/app/reload_unix.go:12-33`

`$PTYLINE_PID` берётся на веру. Устаревшее значение (ptyline умер, PID
переиспользован) приведёт к SIGUSR1 постороннему процессу — а действие по
умолчанию для SIGUSR1 это завершение процесса. Стоит проверять, что
целевой процесс — действительно ptyline (например, сверка имени процесса
или unix-сокет/pidfile вместо сигнала «вслепую»).

### M6. Многострочные команды полностью выпадают из трекинга
`internal/proxy/osc.go:50-57`, шаблоны shell-интеграции

Шаблоны эмитят `command=$cmd` как есть; для многострочной команды значение
содержит `\n`, `hasControlChars` отвергает весь фрейм — событие «команда
началась» теряется (и duration к ней не привяжется). Логичнее
санитизировать значение на стороне шаблона/фильтра (заменять control-байты
пробелом), чем молча отбрасывать.

## Низкая серьёзность

### L1. Переписывание `CSI 2 J` принудительно уводит курсор в home
`internal/proxy/ansi_filter.go:303-318`. Настоящий ED(2) не двигает курсор;
программа, эмитящая `2J` без `CUP`, получит неожиданное позиционирование.
Осознанный компромисс (задокументирован), но остаётся расхождением с VT-семантикой.

### L2. `scanString` не поддерживает 8-битный ST (0x9C) и рвёт DCS с внутренними ESC
`internal/proxy/ansi_filter.go:223-240`. OSC/DCS, терминированные C1 ST,
буферизуются до лимита и выпадают в passthrough; tmux DCS-passthrough
(`ESC P tmux; …ESC ESC…`) преждевременно «закрывается» на внутреннем ESC.

### L3. Парсинг ветки git ломается на именах со `" ["` или `"..."`
`internal/modules/git.go:240-264`. Имя ветки `feat/a...b` или `x [y]`
распарсится неверно. `--porcelain=v2 --branch` даёт машиночитаемый
заголовок без этих неоднозначностей.

### L4. Запросы git-обновления при гонке отбрасываются, а не коалесцируются
`internal/app/app.go:279-297`. Если refresh уже идёт, новый запрос (например
после `cd`) теряется, а результат текущего отбрасывается по `expectedCWD` —
git-блок может показывать данные старого каталога до следующего тика
(до 2 с). Exec-модули решают это через pending-флаг
(`internal/app/exec_runtime.go:83-122`) — тот же паттерн применим здесь.

### L5. ssh-обёртка берёт «последний аргумент» как хост
`templates/bash.sh:115-121`, `zsh.sh:102-108`, `fish.sh:118-124`.
Для `ssh host uptime` или `ssh -L 8080:… host` в бар попадёт не хост.
Косметика, но заметная.

### L6. Комментарий Validate обещает обязательный timeout, проверки нет
`internal/config/loader.go:79` («custom-command modules must carry a
timeout») — фактической проверки `TimeoutMS` в `Validate` нет; молча
применяется дефолт 1 с (`internal/app/exec_runtime.go:36-39`). Либо
проверять, либо поправить комментарий.

### L7. Project-оверлей может расширить `module.X.env` exec-модуля
`internal/config/loader.go:306-331` блокирует command/timeout/source, но
`env` разрешён: `.ptyline` чужого репозитория может заставить зеркалировать
дополнительные (в т.ч. чувствительные) переменные окружения в env
пользовательских exec-команд и в отображаемый `{env}`. Значения не покидают
машину, но границу доверия стоит зафиксировать (запретить или документировать).

### L8. Отложенный из-за rate-limit кадр бара не имеет запланированного дожима
`internal/proxy/writer.go:143-151`. Подавленный лимитером кадр ждёт
следующего события; на практике его будит ежесекундный time-модуль, но при
отключённых интервальных модулях бар может «зависнуть» с устаревшим
содержимым до следующего вывода/тика.

### L9. `FindProjectConfig` статит все родительские каталоги в event loop
`internal/config/loader.go:253-269`, вызывается на каждый OSC `cwd`
(`internal/app/app.go:661-664`). На медленной сетевой ФС это синхронные
stat'ы в горутине loop. Кэш по каталогу или off-loop обход снимут риск.

### L10. При ошибке записи в PTY/терминал дочерняя группа процессов не терминируется
`internal/proxy/eventloop.go:64-93` — `return 1, err` без `Terminate`;
ребёнок остаётся жить и умрёт только по закрытию master-fd при выходе
процесса. Практически безопасно, но путь выхода неоднородный.
